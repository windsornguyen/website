# Parallel Agentic Engineering

<small>Last updated: April 26, 2026 at 8:09 PM PDT.</small>

I’m suspicious that simultaneous AI coding sessions actually boost productivity.

The bottleneck of AI-generated code is the **code reviewing** process. Therefore, if we have more sessions, there is more code to review.

There is also more context switching for the human between sessions. Unless there is a scalable way to reliably automate the code review process, it doesn’t make sense to generate several solutions to a problem all of which require review.

It doesn’t make sense either to solve 3 different problems at once being bottlenecked when you could instead just make sure you solve one singular problem really well with your AI coding tools.

It’s kind of like building a super sleek sports car, except its engine is just a little human under the hood carrying the car manually.

We’re trying to support superhuman processes while being bottlenecked by humans ourselves at the most important part of the entire process (code review). In this regime, net efficiency gains will be scarcer than expected.

What does this mean for the future?

For starters, companies that can nail the correct abstraction and interface to efficiently and reliably review AI-generated code en masse will be wildly successful.

What might this entail? First, the software engineers have to trust that this automated code reviewing system is trustworthy. Does it follow the codebase style guide? Does it follow programming best practices? Has it checked test coverage? Is the code future-proof (composable and maintainable)? The verifiability problem is still largely unsolved, and it’s a really hard problem to get right.

The second requirement is that it has to feel natural. Does it actually feel like a senior engineer reviewing the code? Does it write like a human and not like an LLM? Is the PR bloated with inscrutable paragraphs of comments? Can I easily ping the code review agent either via Slack or some other centralized communications software?

It should not feel like I’m delegating to the machine but rather to my colleague. And I should trust that my colleague does a good job at reviewing the code (ironically, that they probably generated).

The third requirement is the knowledge transfer process. This one is not discussed as much, but it is vital for code maintainability. If AI generates a codebase that’s 1M lines of code, and you’ve delegated code review entirely to another agent, you’re going to be in for a hard time when something inevitably breaks in production and now agents with different context windows try to solve the problem.

What needs to happen instead is the code generated by AI has to be well-documented and easily readable by humans. If we live in a world where AI is writing all of our code and simultaneously reviewing all of the code, this step of manual knowledge transfer—where the human has to read the code written by the AI—becomes the bottleneck. Unfortunately, it does not seem to be avoidable if the human wants to actually deeply understand the artifacts being pumped out by the AI.

If humans want to have an intellectual stake in the code being produced, it is likely that parallel agentic engineering will never truly take off.

Note that this does NOT mean it is impossible to have fully autonomous agentic coding systems. In fact, that is probably going to happen before 2028. It DOES mean, though, that this system by definition will have to be decoupled from the human. It is up to society at that point to deem whether this is acceptable.

Historical precedent tells us this will likely not be societally accepted.
